# 俱乐部成员专用小程序 - 产品需求文档（PRD）

**版本号：** v1.0  
**文档日期：** 2024  
**产品名称：** 俱乐部小程序  
**技术栈：** 微信小程序（原生）+ 微信云开发（CloudBase）

---

## 一、产品概述

### 1.1 产品定位
一款面向俱乐部内部成员的小程序，提供活动报名登记和AA记账结算功能，帮助成员高效管理活动参与和费用分摊。

### 1.2 核心价值
- **权限控制**：通过邀请码机制确保仅内部成员可访问
- **活动管理**：统一登记活动参与成员，便于后续记账时自动关联
- **智能记账**：基于日期和报名数据，自动计算人均费用和结算明细
- **数据持久化**：使用微信云开发，无需自建服务器，数据自动同步

### 1.3 目标用户
俱乐部内部成员（需持有邀请码）

## 二、功能模块详细说明

### 2.1 权限验证模块

#### 2.1.1 功能描述
用户首次进入小程序时，必须输入正确的邀请码才能访问后续功能。

#### 2.1.2 交互流程
1. 用户打开小程序，显示邀请码输入界面
2. 用户输入邀请码（密码模式，隐藏显示）
3. 点击“进入系统”按钮
4. 系统校验邀请码：
   - **成功**：显示主功能界面，加载历史数据
   - **失败**：提示“邀请码错误，无权访问”，停留在验证界面

#### 2.1.3 业务规则
- **固定邀请码**：`dragon`（硬编码在代码中）
- **验证状态**：通过验证后，状态保存在 `session_state.authenticated`，本次会话有效
- **安全机制**：输入框使用 `password="true"` 隐藏输入内容

#### 2.1.4 UI元素
- 标题：`俱乐部成员专用小程序`
- 副标题：`请输入内部邀请码`
- 输入框：密码类型，占位符 `请输入邀请码`
- 按钮：`进入系统`（主色调）
- 提示文字：`邀请码错误将无法访问后续内容`

### 2.2 活动报名模块

#### 2.2.1 功能描述
成员可登记参与某日活动的成员姓名，数据保存到云数据库，供后续AA记账时自动关联。

#### 2.2.2 数据输入
- **活动日期**：`picker mode="date"`，格式 `YYYY-MM-DD`
- **参与成员姓名**：文本输入框，支持中文、英文、数字

#### 2.2.3 提交逻辑
1. **前端校验**：
   - 日期必填，未选择时提示“请选择日期”
   - 姓名必填且不能为空字符串，未填写时提示“请输入姓名”
2. **数据保存**：
   - 调用云数据库 `activities` 集合的 `add` 方法
   - 保存字段：`date`（活动日期）、`name`（成员姓名）、`createdAt`（服务器时间）
3. **反馈**：
   - 成功：显示 Toast “报名成功”，清空姓名输入框，刷新历史报名列表
   - 失败：显示 Toast “提交失败”，保留输入内容

#### 2.2.4 历史报名展示
- **数据来源**：云数据库 `activities` 集合，按 `createdAt` 降序排列
- **展示内容**：
  - 日期 - 姓名（主标题）
  - 提交时间（副标题，格式：`YYYY-MM-DD HH:mm:ss`）
- **展示位置**：活动报名模块下方，用分隔线区分

#### 2.2.5 业务规则
- 同一日期可多次提交不同成员（支持多人报名）
- 同一成员可在同一日期多次报名（系统不做去重，保留所有记录）
- 日期格式统一为 `YYYY-MM-DD`

### 2.3 AA记账模块（核心功能）

#### 2.3.1 功能描述
基于日期和活动报名数据，记录支出账单，自动计算人均费用和结算明细，并提供当日汇总视图。

#### 2.3.2 日期驱动逻辑

**2.3.2.1 日期选择器**
- **位置**：AA记账模块最顶部
- **组件**：`picker mode="date"`
- **字段**：`aaDate`（格式：`YYYY-MM-DD`）
- **事件**：`onAADateChange`

**2.3.2.2 动态成员加载**
- **触发时机**：用户选择日期后立即执行
- **查询逻辑**：
  ```javascript
  db.collection("activities")
    .where({ date: selectedDate })
    .get()
  ```
- **数据处理**：
  - 从查询结果中提取所有 `name` 字段
  - 去重（使用对象 `nameSet` 去重）
  - 转换为 `aaMembers` 数组，格式：
    ```javascript
    [
      { name: "成员A", checked: true },
      { name: "成员B", checked: true },
      ...
    ]
    ```
  - 默认所有成员 `checked: true`（全选状态）
- **状态更新**：
  - 清空 `aaPayerIndex`、`aaPayerName`、`settlements`、`dailySummary`
  - 如果成员列表非空，默认选择第一个成员作为付款人

**2.3.2.3 成员列表为空处理**
- 如果该日期无报名记录，显示提示：`该日期暂无报名成员，请先在"活动报名"页登记。`
- 此时付款人和参与人选择器不可用

#### 2.3.3 付款人选择（单选）

**2.3.3.1 组件类型**
- `picker mode="selector"`
- 数据源：`aaMembers` 数组
- `range-key="name"`（显示成员姓名）

**2.3.3.2 交互逻辑**
- 用户从下拉列表中选择一个成员
- 选择后触发 `onAAPayerChange`：
  - 更新 `aaPayerIndex`（选中索引）
  - 更新 `aaPayerName`（选中成员姓名）
  - **强制逻辑**：确保付款人在参与人列表中（如果未勾选，自动勾选）

**2.3.3.3 业务规则**
- 付款人必须从当日报名成员中选择（不能手动输入）
- 付款人必须包含在参与人列表中（系统自动保证）

#### 2.3.4 参与人选择（多选）

**2.3.4.1 组件类型**
- `checkbox-group` + `checkbox`
- 数据源：`aaMembers` 数组，每个成员渲染一个 `checkbox`

**2.3.4.2 交互逻辑**
- 用户可勾选/取消勾选多个成员
- 选择变化触发 `onAAParticipantsChange`：
  - 根据选中的 `value` 数组更新 `aaMembers[*].checked`
  - **强制逻辑**：如果付款人未被勾选，系统自动勾选付款人

**2.3.4.3 全选功能**
- 提供“全选”按钮（`onSelectAllParticipants`）
- 点击后，将所有 `aaMembers[*].checked` 设为 `true`

**2.3.4.4 业务规则**
- 至少选择一位参与人（提交时校验）
- 付款人始终在参与人列表中（系统保证）

#### 2.3.5 支出信息输入

**2.3.5.1 支出事项**
- 文本输入框
- 占位符：`例如：聚餐、房费、门票等`
- 字段：`billItem`

**2.3.5.2 支出总金额**
- 数字输入框（`type="digit"`）
- 占位符：`例如：300`
- 字段：`billAmount`
- 支持小数（如：`299.50`）

#### 2.3.6 提交与计算逻辑

**2.3.6.1 前端校验**
按顺序校验以下条件，任一不满足则中断并提示：
1. **记账日期**：必填，未选择提示“请选择记账日期”
2. **支出事项**：必填且不能为空字符串，未填写提示“请填写支出事项”
3. **金额**：必须为有效数字且 > 0，不满足提示“金额需大于 0”
4. **成员列表**：`aaMembers.length > 0`，为空提示“该日期暂无报名成员，无法记账”
5. **付款人**：`aaPayerIndex >= 0`，未选择提示“请选择付款人”
6. **参与人**：至少勾选一位，未勾选提示“请至少选择一位参与人”

**2.3.6.2 计算逻辑**
```javascript
// 1. 提取参与人姓名列表
participants = aaMembers.filter(m => m.checked).map(m => m.name)

// 2. 计算人均金额
count = participants.length
perShare = parseFloat((totalAmount / count).toFixed(2))

// 3. 生成结算明细（本次账单）
settlements = participants
  .filter(p => p !== payer)
  .map(p => ({
    from: p,        // 付款人（应付）
    to: payer,      // 收款人（应收）
    amount: perShare
  }))
```

**2.3.6.3 数据保存**
保存到云数据库 `bills` 集合，字段结构：
```javascript
{
  date: "2024-01-15",           // 记账日期（关联活动日期）
  item: "聚餐",                  // 支出事项
  totalAmount: 300,              // 总金额（数字）
  payer: "张三",                 // 付款人姓名
  participants: ["张三", "李四", "王五"],  // 参与人姓名数组
  perShare: 100,                 // 人均金额（数字，保留2位小数）
  createdAt: ServerDate()        // 服务器时间
}
```

**2.3.6.4 提交后处理**
- 成功：显示 Toast “记账成功”，清空 `billItem`、`billAmount`，保留 `settlements` 用于展示，刷新历史账单列表，重新加载当日汇总
- 失败：显示 Toast “保存失败”，保留输入内容

#### 2.3.7 本次结算结果展示

**2.3.7.1 展示时机**
- 提交账单成功后，在“本次结算结果”区域展示
- 数据来源：`settlements` 数组

**2.3.7.2 展示格式**
```
成员A → 付款人：100 元
成员B → 付款人：100 元
...
```

**2.3.7.3 空状态**
- 未提交账单或 `settlements.length === 0` 时，显示：`提交一条账单后，这里会显示"谁给谁多少钱"`

#### 2.3.8 当日结算单（日期维度汇总）

**2.3.8.1 功能描述**
根据用户选择的记账日期，自动汇总该日期下所有账单，计算每个成员的净应收/应付金额。

**2.3.8.2 数据加载**
- **触发时机**：
  - 用户选择记账日期时（`onAADateChange`）
  - 提交账单成功后
- **查询逻辑**：
  ```javascript
  db.collection("bills")
    .where({ date: selectedDate })
    .get()
  ```

**2.3.8.3 汇总计算逻辑（`buildDailySummary`）**

**步骤1：遍历所有账单，统计每个成员的数据**
```javascript
stats = {}  // 格式：{ "成员名": { paid: 0, shouldPay: 0 } }

for each bill in bills:
  totalAmount += bill.totalAmount
  
  // 统计每个参与人的应付金额
  for each participant in bill.participants:
    stats[participant].shouldPay += bill.perShare
  
  // 统计付款人的垫付金额
  stats[bill.payer].paid += bill.totalAmount
```

**步骤2：计算净额**
```javascript
for each member in stats:
  paid = stats[member].paid          // 累计垫付金额
  shouldPay = stats[member].shouldPay  // 累计应付金额
  net = paid - shouldPay              // 净额
  
  // 生成展示文本
  if net > 0:
    summaryText = "垫付 {paid} 元，应付 {shouldPay} 元，净应收 {net} 元"
  else if net < 0:
    summaryText = "垫付 {paid} 元，应付 {shouldPay} 元，净应付 {Math.abs(net)} 元"
  else:
    summaryText = "垫付 {paid} 元，应付 {shouldPay} 元，已结清"
```

**步骤3：返回汇总结果**
```javascript
{
  date: "2024-01-15",
  totalAmount: 600,  // 当日总支出
  members: [
    {
      name: "张三",
      paid: 300,
      shouldPay: 200,
      net: 100,
      summaryText: "垫付 300 元，应付 200 元，净应收 100 元"
    },
    {
      name: "李四",
      paid: 0,
      shouldPay: 200,
      net: -200,
      summaryText: "垫付 0 元，应付 200 元，净应付 200 元"
    },
    ...
  ]
}
```

**2.3.8.4 UI展示**
- **位置**：AA记账模块底部，用分隔线与“本次结算结果”区分
- **标题**：`当日结算单`
- **展示内容**：
  - 顶部：`日期：{date}`、`总支出：{totalAmount} 元`
  - 列表：每个成员一行，显示 `{name}` 和 `{summaryText}`
- **空状态**：
  - 未选择日期：`请选择记账日期后查看当日汇总。`
  - 无账单记录：`该日期暂无记账记录。`

**2.3.8.5 业务规则**
- 汇总范围：仅限用户选择的记账日期
- 金额精度：所有金额保留2位小数
- 成员排序：按成员姓名自然排序（Object.keys 默认排序）

## 三、数据库表结构

### 3.1 云开发环境配置
- **环境ID**：`cloudbase-0grjcjxkb55037a3`
- **数据库类型**：微信云开发 NoSQL 数据库
- **权限设置**：建议使用“仅创建者可读写”（可根据需求调整）

### 3.2 activities 集合（活动报名）

#### 3.2.1 集合名称
`activities`

#### 3.2.2 字段结构
| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| `_id` | String | 是（自动） | 文档唯一标识 | `"abc123..."` |
| `date` | String | 是 | 活动日期，格式 `YYYY-MM-DD` | `"2024-01-15"` |
| `name` | String | 是 | 参与成员姓名 | `"张三"` |
| `createdAt` | Date | 是 | 服务器时间（创建时间） | `ServerDate()` |

#### 3.2.3 索引建议
- **查询索引**：`date`（用于按日期查询报名成员）
- **排序索引**：`createdAt`（用于历史记录倒序展示）

#### 3.2.4 数据示例
```json
{
  "_id": "abc123def456",
  "date": "2024-01-15",
  "name": "张三",
  "createdAt": {
    "$date": "2024-01-15T10:30:00.000Z"
  }
}
```

### 3.3 bills 集合（AA账单）

#### 3.3.1 集合名称
`bills`

#### 3.3.2 字段结构
| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| `_id` | String | 是（自动） | 文档唯一标识 | `"xyz789..."` |
| `date` | String | 是 | 记账日期（关联活动日期），格式 `YYYY-MM-DD` | `"2024-01-15"` |
| `item` | String | 是 | 支出事项 | `"聚餐"` |
| `totalAmount` | Number | 是 | 支出总金额（保留2位小数） | `300.00` |
| `payer` | String | 是 | 付款人姓名 | `"张三"` |
| `participants` | Array<String> | 是 | 参与人姓名数组 | `["张三", "李四", "王五"]` |
| `perShare` | Number | 是 | 人均金额（保留2位小数） | `100.00` |
| `createdAt` | Date | 是 | 服务器时间（创建时间） | `ServerDate()` |

#### 3.3.3 索引建议
- **查询索引**：`date`（用于按日期查询账单，支持当日汇总）
- **排序索引**：`createdAt`（用于历史记录倒序展示）

#### 3.3.4 数据示例
```json
{
  "_id": "xyz789abc123",
  "date": "2024-01-15",
  "item": "聚餐",
  "totalAmount": 300,
  "payer": "张三",
  "participants": ["张三", "李四", "王五"],
  "perShare": 100,
  "createdAt": {
    "$date": "2024-01-15T14:20:00.000Z"
  }
}
```

#### 3.3.5 数据约束
- `totalAmount > 0`
- `participants.length > 0`
- `payer` 必须在 `participants` 数组中
- `perShare = totalAmount / participants.length`（保留2位小数）

## 四、交互流程

### 4.1 首次使用流程
```
用户打开小程序
  ↓
显示邀请码输入界面
  ↓
用户输入邀请码 "dragon"
  ↓
点击"进入系统"
  ↓
系统校验邀请码
  ├─ 成功 → 显示主功能界面（默认显示"活动报名"Tab）
  └─ 失败 → 提示"邀请码错误"，停留在验证界面
```

### 4.2 活动报名流程
```
用户切换到"活动报名"Tab
  ↓
选择活动日期（picker）
  ↓
输入参与成员姓名
  ↓
点击"提交报名"
  ↓
前端校验（日期、姓名必填）
```

```
用户切换到"AA 记日记"Tab

选择记账日期（picker）

      选择付款人（picker，从成员列表中选择）
```

```
  ├─ 活动报名：查询 activities 集合，按 createdAt 降序
  └─ AA账单：查询 bills 集合，按 createdAt 降序
  ↓
展示历史记录列表
```

## 五、业务逻辑详解

### 5.1 日期关联逻辑
- **活动报名**：成员在某个日期报名，数据保存在 `activities` 集合，字段 `date`
- **AA记账**：记账时必须先选择日期，系统从 `activities` 集合查询该日期的所有报名成员
- **数据一致性**：记账的 `date` 字段必须与活动报名的 `date` 字段一致，才能正确关联成员

### 5.2 成员去重逻辑
- **报名阶段**：不做去重，同一成员可在同一日期多次报名（保留所有记录）
- **记账阶段**：从 `activities` 查询时，使用对象 `nameSet` 去重，确保成员列表不重复

### 5.3 付款人与参与人关联逻辑
- **强制关联**：付款人必须包含在参与人列表中
- **自动勾选**：
  - 选择付款人时，如果付款人未勾选，系统自动勾选
  - 取消参与人时，如果取消的是付款人，系统自动重新勾选付款人

### 5.4 金额计算逻辑
- **人均金额**：`perShare = totalAmount / participants.length`，保留2位小数
- **结算明细**：除付款人外，每个参与人向付款人支付 `perShare` 元
- **精度处理**：使用 `parseFloat((amount / count).toFixed(2))` 确保精度

### 5.5 当日汇总计算逻辑
- **统计维度**：按记账日期（`date`）汇总
- **成员统计**：
  - `paid`：该成员作为付款人时，所有账单的 `totalAmount` 之和
  - `shouldPay`：该成员作为参与人时，所有账单的 `perShare` 之和
- **净额计算**：`net = paid - shouldPay`
  - `net > 0`：净应收（垫付多了，应收回钱）
  - `net < 0`：净应付（该付的钱还没付够）
  - `net = 0`：已结清
- **总支出**：该日期下所有账单的 `totalAmount` 之和

### 5.6 数据同步逻辑
- **实时加载**：
  - 选择记账日期时，立即查询成员和当日账单
  - 提交账单成功后，立即刷新历史列表和当日汇总
- **历史记录**：切换到“历史记录”Tab 时，重新查询最新数据
